§2: Physical memory inventory [1 defect]
IDSeverityDescriptionSuggested fixFPL-01BField Device QSPI: LCD assets Size column says "1 MB" but the stated addresses give 960 KB. Address arithmetic: 0x900F_FFFF − 0x9001_0000 + 1 = 0x000F_0000 = 983 040 B = 960 KB. §1 declares addresses authoritative; the Size column contradicts them. The 16 MB total closes correctly because the Reserved gains the 64 KB difference silently. An LLD engineer reading the Size column will allocate 1 MB for LVGL assets and overrun the partition boundary.Change the Size cell to 960 KB (or move the end address to 0x9010_FFFF and adjust Reserved start to 0x9011_0000, accepting that Reserved shrinks to ~14 MB — but verify the total still closes).

§5: Gateway partition layout [2 defects]
IDSeverityDescriptionSuggested fixFPL-02BGateway QSPI has no X.509 certificate partition, violating CON-006. CON-006: "X.509 client certificates and private keys shall be stored in a dedicated flash partition, separate from configuration and telemetry buffer storage." The QSPI layout contains ConfigStore (configuration), CircularFlashLog (telemetry/log buffer), OTA staging, and Reserved — but no certificate partition. Certificates cannot go into ConfigStore (that is the configuration storage) or CircularFlashLog (that is the telemetry buffer storage).Carve a dedicated certificate partition (e.g., 64 KB) out of the Reserved region, assign a fixed address range, and cite CON-006 and REQ-NF-302.FPL-07FCircularFlashLog partition is described only as "Diagnostic log ring buffer (REQ-NF-500 series)" but components.md shows StoreAndForward uses CircularFlashLog as its backing store, making this partition serve dual duty: diagnostic logging and telemetry buffering during cloud outages. REQ-BF-000 requires non-volatile storage for buffering; it is not cited here. The 1 MB partition size is justified only for logging; the additional write pressure and storage budget from buffered telemetry (governed by REQ-NF-407) are not addressed.Expand the Purpose cell to acknowledge both uses; add REQ-BF-000–REQ-BF-020 and REQ-NF-407 to the requirement citations; extend the ring-size justification in §6.2 to cover both workloads.

§6: Wear-levelling strategy [3 defects]
IDSeverityDescriptionSuggested fixFPL-03B§6.2 ring-size justification is arithmetically wrong by a factor of 111×. The document states: "At a representative log rate of one record per second averaging 64 bytes, the 1 MB ring offers approximately three weeks of history before wrap." Verified: 1 048 576 B ÷ 64 B/s = 16 384 s ≈ 4.55 hours, not three weeks (1 814 400 s). A 110.7 MB ring would be needed for three weeks at the stated rate. This is the sole ring-size justification; it is numerically unsound and untestable as written.Either correct the arithmetic (4.55 h fill at 1 rec/s × 64 B) and re-justify the 1 MB allocation on a different basis (e.g., a lower realistic diagnostic log rate such as ~1 event per 2 min), or change the ring size to match the three-week target. State the assumed log rate explicitly so it can be validated during integration.FPL-05F§6.1 ConfigStore slot-selection criterion is incompletely specified for the power-loss-safe claim. §6.1 says: "The config-load logic reads both slots on boot and selects the one with the higher sequence number." If power fails after the sequence number of the new slot is written but before the CRC is written (or while data bytes are still being written), the new slot will have a higher sequence number but an invalid CRC. The stated selection criterion would pick the corrupt slot. The document claims power-loss safety but does not state that CRC validity is a prerequisite for slot selection.Amend the load-protocol description to: "selects the slot with the highest sequence number whose CRC32 is valid"; and note that the sequence number must be the last field written in a slot commit.FPL-08FCON-009 endurance budget is not demonstrated numerically for CircularFlashLog — the most-write-hot partition. §6 states the endurance concern qualitatively; §6.2 gives a log rate but never derives the erase-cycle rate per sector or compares it against CON-009's 100 000-cycle limit. Verified (using the document's own rate assumption before the arithmetic is corrected): at 1 rec/s × 64 B, each 4 KB sector fills in 64 s; the 256-sector ring revolves every 16 384 s (4.55 h), giving ≈ 1 925 erases/sector/year → 52-year flash lifetime, which is compliant. This should be shown in the document. Once FPL-03 is fixed and the true log rate is established, the budget must be re-derived.Add a dedicated endurance budget sub-section under §6.2: state assumed log rate, derive sector fill time, derive erase rate, compare against CON-009, state device-lifetime figure.

§7: Bootloader contract [2 defects]
IDSeverityDescriptionSuggested fixFPL-10F§7.1 metadata offset table does not reflect the A/B sector rotation structure asserted in §6.4 and in the final paragraph of §7.1. §6.4 states "The 8 KB / 4-sector budget allows A/B sector rotation for the most frequently updated fields (active bank, pending_self_check flag, rollback count)"; §7.1 then says "The bootloader uses A/B rotation across the metadata partition's sectors for the frequently updated fields." However, the §7.1 offset table shows a single flat structure at 0x0000–0x008F+ with no sector-level A/B split, no slot-A vs slot-B address ranges, and no sequence number field for the rotation logic. §1 declares these addresses authoritative. The LLD bootloader developer cannot derive the correct layout from this table.Add a second offset sub-table (or annotation) that shows the A-copy vs B-copy sector boundaries and the per-copy field layout, including the sequence/generation counter that governs which copy is the active one.FPL-06F§7 does not specify the mechanism by which application firmware triggers a rollback after a failed self-check. §7 step 3 says: "the firmware is then responsible for … triggering a rollback on failure" without saying how. The bootloader cannot implement the rollback path without knowing the trigger protocol (e.g., reset while pending_self_check remains set, or write a rollback_request flag in metadata, or decrement a retry counter). This is part of the bootloader–firmware ABI and belongs in the HLD contract.Add a step 3a to §7 specifying the trigger protocol, e.g.: "Firmware signals a rollback by performing a software reset without clearing pending_self_check. On the subsequent boot, if pending_self_check is set and the rollback count in metadata equals a configurable maximum, the bootloader switches the active bank and reboots again." Align with the Firmware Update state machine in state-machines.md.

§8: Constraints satisfied [1 defect]
IDSeverityDescriptionSuggested fixFPL-04B§8 claims StoreAndForward is "Handled in RAM" — this is factually wrong and contradicts two reference documents. The table reads: *"REQ-BF-000..-020Handled in RAM by StoreAndForward; flash partition not required for transient buffering."* (1) components.md explicitly lists StoreAndForward USES (downward): CircularFlashLog, IHealthReport, ILogger — CircularFlashLog is QSPI-flash-backed. (2) SRS REQ-BF-000 mandates "non-volatile storage" for outbound cloud message buffering. RAM is volatile; an in-RAM buffer would lose all buffered messages on power-cycle, violating REQ-BF-000. This misstatement also conceals that CircularFlashLog serves dual duty (see FPL-07).

Summary
Top three blockers: FPL-03 undermines the entire ring-size rationale for CircularFlashLog — the "three weeks" figure is 111× wrong, making the partition-sizing justification numerically unsound and untestable as written. FPL-02 is a hard constraint violation: CON-006 mandates a dedicated X.509 certificate partition separate from configuration and telemetry storage, but no such partition exists in the Gateway QSPI layout. FPL-04 is a cross-document contradiction with cascading consequences: claiming StoreAndForward operates in RAM misrepresents the system's durability guarantee, contradicts components.md, and hides the dual-purpose nature of the CircularFlashLog partition.
Overall posture: The Gateway on-chip address arithmetic is sound (the 1 MB ✓ sum in §5.1 checks out). The bootloader address constants (0x0800_6000 Bank A, 0x0807_E000 Bank B) are correct, the OTA staging sizing and resumable-download rationale are well-argued, and the A/B rotation strategy for ConfigStore is conceptually correct — only its load-protocol description is underspecified (FPL-05). The field device layout is simple and uncontroversial; the sole defect there is a Size label typo (FPL-01, 960 KB vs. 1 MB). The most systematic weakness is in §8: the requirements-to-partitions traceability table has one factual error (FPL-04) and is silent on the certificate constraint (FPL-02) and the store-and-forward use of CircularFlashLog (FPL-07).
Recommendation: do not advance to LLD. The four blockers (FPL-01 through FPL-04) must be resolved and the document re-reviewed before a Phase 3 gate can be passed. In particular, FPL-03 requires the ring-size justification to be rebuilt from a validated log-rate assumption, and that assumption must then feed the CON-009 endurance budget (FPL-08) before the document can be called numerically sound.
