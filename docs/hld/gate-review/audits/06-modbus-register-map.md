§2: Protocol parameters [2 defects]
IDSeverityDescriptionSuggested fixRM-01BWrong requirement citation for all four serial parameters. Baud rate, data bits, parity, and stop bits all cite REQ-MB-010. REQ-MB-010 is "The system shall read sensor data from the field device via Modbus read input registers" — a functional read requirement, not a protocol-parameter requirement. The correct reference is REQ-MB-030 ("The system shall communicate via Modbus RTU at: Baudrate 9600, Data bits 8, Parity no parity, Stop bits 1"). Every integration test that verifies §2 against the SRS would trace to the wrong requirement and fail review.Replace REQ-MB-010 with REQ-MB-030 in all four rows of the parameter table.RM-02FREQ-MB-050 and REQ-MB-060 cited under "Response timeout" and "Retry count" without the companion NF trace. REQ-NF-105 independently mandates the 200 ms budget; REQ-NF-103/NF-104 define the 3-consecutive-failure/success hysteresis that determines when REQ-MB-060's retry limit triggers link-down. Omitting these leaves the timing requirement partially untraceable.Add REQ-NF-105 alongside REQ-MB-050 and REQ-NF-103 alongside REQ-MB-060 in the reference column.

§3: Function code support — clean.
(The FC table is consistent with SRS REQ-MB-040; the exception-per-FC gap is flagged in §7 below where the exception table lives.)

§4: Address-space layout [1 defect]
IDSeverityDescriptionSuggested fixRM-03BReserved-address read behaviour contradicted internally. §4 states: "Read attempts on reserved addresses return exception 0x02 — Illegal Data Address." But §6.1 states 0x0008–0x000F: "Returns 0 on read", and §6.4's reserved config block (0x0150–0x01FF) explicitly states "Read returns 0." Three distinct behaviours are asserted for the same class of address, with no rule distinguishing them. This makes conformance testing impossible: a slave that returns 0 on TC-RM-021 would fail §4 but pass §6.4.Adopt one canonical rule and state it once. Industry norm for Modbus: intra-section reserved registers return 0 (they share a valid FC); inter-section gaps return exception 0x02 (no valid FC maps there). State this split explicitly in §4 and remove the conflicting per-section clauses.

§5: Encoding conventions — clean.
(Big-endian ABCD word order documented, §5.1. Sentinel values defined per type, §5.4. Scaled-integer rationale stated, §5.3. All three specialisation checks satisfied.)

§6: Register definitions [3 defects]
IDSeverityDescriptionSuggested fixRM-04BDEVICE_STATE enum values 3 and 4 have no matching states in the Field Device Lifecycle state machine. The register map encodes 3=Error, 4=Shutdown. state-machines.md Machine 5 (Field Device Lifecycle) defines four states: Init, Operational, EditingConfig, Faulted. There is no "Error" state and no "Shutdown" state. Value 3 is a name mismatch; value 4 refers to a state that does not exist in the machine. A gateway reading DEVICE_STATE == 4 cannot map it to any defined lifecycle state. P6 violation: register behaviour has no traceable machine state.Rename value 3 to Faulted to match state-machines.md. Remove value 4 (Shutdown) or add a Shutdown state to the FD Lifecycle machine with a corresponding state-machine transition.RM-05BSAMPLING_PERIOD_MS (0x0130) default value (100 ms) contradicts two SRS requirements. REQ-SA-010 states "default polling interval of 1 second if the configuration reading fails"; REQ-NF-110 states "default polling rate of 1 Hz." The register default is 100 ms (10 Hz). A freshly flashed field device will sample at ten times the specified default rate on first boot.Change default to 1000 (1000 ms = 1 Hz), matching REQ-SA-010 and REQ-NF-110.RM-06FSource/sink components are absent or inconsistent across nearly all register entries. The audit specialisation check requires every register to document both source component and sink component. A small subset mentions a source in free text (e.g., TEMPERATURE: "from HumidityTempDriver"; DEVICE_STATE: "Source: LifecycleController"). No register documents a sink. Most registers state neither. HARDWARE_REV, FW_VERSION_*, VENDOR_CODE, ALARM_FLAGS, UPTIME_SECONDS, all Modbus/sensor error counters, all alarm-threshold config registers, and all LCD config registers have no component attribution at all.Add dedicated Source and Sink columns to every register table. Populate systematically: sensor reading sources are the relevant driver; state-metric sources are LifecycleController / IModbusSlaveStats as appropriate; config register sinks are ConfigService; command register sinks are the named action handler.
§6.x (cross-cutting): Missing requirement citations [1 defect]
IDSeverityDescriptionSuggested fixRM-07FThe majority of registers carry no requirement citation. The audit specialisation check states "Every register cites the requirement (REQ-MB / REQ-SA / REQ-DM) that motivates it." The following registers are entirely uncited: HARDWARE_REV (0x0003); FW_VERSION_MAJOR/MINOR/PATCH (0x0004–0x0006); VENDOR_CODE (0x0007); TEMPERATURE, HUMIDITY, PRESSURE (0x0010–0x0012); ALARM_FLAGS (0x0031); UPTIME_SECONDS (0x0032–0x0033); all four Modbus/sensor counter pairs (0x0034–0x003B); all alarm-threshold configuration registers (0x0100–0x0122); LCD_BRIGHTNESS_PCT, LCD_TIMEOUT_S (0x0140–0x0141); CMD_RESET_METRICS (0x0201). The single cited exception, SAMPLING_PERIOD_MS → REQ-SA-070, is correct. Uncited registers violate P6 (Responsibility traces to requirements): a gate reviewer cannot confirm the register is contractually necessary.For each uncited register add at minimum one REQ-* citation in the Description column. Guidance: identity registers → REQ-MB-111; sensor readings → REQ-SA-070 / REQ-MB-010; alarm flags → REQ-AM-020; uptime → REQ-LD-070 (displayed on status screen); Modbus counters → REQ-LD-070; alarm thresholds → REQ-AM-000/REQ-AM-011; LCD config → REQ-LD-050/REQ-LD-100.
§6.4: Configuration — additional defect
IDSeverityDescriptionSuggested fixRM-08FCross-register threshold validation rule not stated. Nothing in §6.4 specifies the exception returned when TEMP_ALARM_LOW >= TEMP_ALARM_HIGH (or the humidity/pressure equivalents) are written. Each threshold register individually documents a range (e.g., −4000..8500), but there is no stated inter-register constraint. A write of TEMP_ALARM_LOW = 7000, TEMP_ALARM_HIGH = 5000 passes each register's individual range check but creates an inverted alarm band that would never fire. Untestable as written (no test case covers this).Add a note to the temperature, humidity, and pressure threshold sub-sections stating the cross-register invariant (e.g., "LOW < HIGH is required; violation of this invariant returns exception 0x03 on the write that creates the inversion") and add a corresponding test case.

§6.5: Commands and control [2 defects]
IDSeverityDescriptionSuggested fixRM-09BCMD_ACK_ALARM (0x0200) cites REQ-AM-040, which does not motivate this register. REQ-AM-040 states "The system shall include in each alarm notification: sensor ID, alarm type (high/low), measured value, threshold value, timestamp, and device ID." This is a payload-content requirement for cloud notifications — it says nothing about acknowledging an alarm via Modbus. No SRS requirement in the [AM] section covers Modbus-side alarm acknowledgement. This is a P6 violation (command without traceable requirement) and a factually wrong citation.Identify or add an SRS requirement covering alarm acknowledgement via Modbus (e.g., derived from UC-09 or UC-08). Replace the REQ-AM-040 citation with the correct reference. Until an SRS requirement exists, this command is architecturally unmotivated.RM-10FFC16 (Write Multiple Registers) on the command range (0x0200–0x02FF) is permitted by §4 but its multi-command behaviour is unspecified. A single FC16 transaction could write both CMD_ACK_ALARM (0x0200) and CMD_SOFT_RESTART (0x0202) in one frame. The document does not state whether commands execute in address order, whether the request is rejected if it spans more than one command register, or whether partial execution is possible. Unspecified behaviour in a safety-adjacent path (soft restart) is untestable.Add a rule to §6.5 stating the policy for FC16 spanning multiple command registers — e.g., "An FC16 write that spans more than one command register returns exception 0x03 — Illegal Data Value. The master must use FC06 to write each command register individually."

§7: Exception responses [1 defect]
IDSeverityDescriptionSuggested fixRM-11FException codes are not mapped per function code. The audit specialisation check states "Exception responses defined for each FC." §7 provides a single global table applicable to all FCs. In practice, 0x04 Slave Device Failure can only arise during FC04 register reads (sensor I/O error) and FC06/FC16 writes (flash fault); it cannot arise during a pure FC03 read of a static config register. 0x03 Illegal Data Value only arises on FC06/FC16 writes. A master implementor reading §7 cannot determine which exceptions to handle per FC without inference. The test suite has no FC-scoped exception test cases.Add a per-FC exception applicability note to §7, e.g., a secondary table with columns FC03 / FC04 / FC06 / FC16 and tick marks per exception code.

§8: Versioning and compatibility [1 defect]
IDSeverityDescriptionSuggested fixRM-12FThe Gateway's "supported set" of MAP_VERSION values is never defined. §8 states: "Slaves reporting a version outside the Gateway's supported set are rejected." §6.1 notes: "mismatch outside supported range is fatal." Neither the current supported version nor the supported range/list is stated anywhere in the document. A firmware team building the gateway cannot implement the check; an integration tester cannot write a passing test for it.State the current supported set explicitly: e.g., "The Gateway firmware v1.x supports MAP_VERSION = 1 only. Any slave returning a different value is rejected during link establishment." Update this clause whenever MAP_VERSION is bumped, alongside the change log.

§10: Test plan [1 defect]
IDSeverityDescriptionSuggested fixRM-13FTC-RM-030 cites REQ-MB-050 as the timing criterion for soft-restart verification. REQ-MB-050 is the Modbus 200 ms request-timeout requirement — it applies to the master waiting for a protocol response, not to device restart recovery. After CMD_SOFT_RESTART, the field device reboots and will not respond within 200 ms; the master will see a timeout, which is expected and correct. The restart recovery budget is REQ-NF-202/REQ-NF-203 (resume within 5 seconds after reset). The test as written would always "fail" its own criterion.Replace "within REQ-MB-050 budget plus boot time" with "within the 5-second budget mandated by REQ-NF-202/REQ-NF-203." Clarify that the master expects no Modbus response to the restart write and should poll MAP_VERSION periodically (≤ 5 s) to detect the device's return to operation.

§4 (address-space rationale): [1 defect]

Summary
Top three blockers:

RM-01 (§2) — REQ-MB-010 is cited as the authority for the serial line parameters (baud, bits, parity, stop). REQ-MB-010 is a functional read requirement, not a protocol-parameter requirement. The correct reference is REQ-MB-030. Every conformance test tracing §2 to the SRS will hit the wrong requirement.
RM-03 (§4 vs §6.1/§6.4) — Three incompatible behaviours are asserted for reads of reserved addresses: exception 0x02 (§4), return 0 (§6.1 identity sub-section), and return 0 (§6.4 config sub-section). A slave implementation must pick one; an integration test cannot pass all three assertions simultaneously.
RM-04/RM-09 (§6.3 and §6.5) — DEVICE_STATE encodes enum values that do not correspond to states in the Field Device Lifecycle state machine (value 3 is named "Error" where the machine uses "Faulted"; value 4 "Shutdown" does not exist in the machine at all). Simultaneously, CMD_ACK_ALARM cites a requirement about alarm-notification payload content as justification for a Modbus acknowledgement command — no SRS requirement motivates alarm ACK via Modbus.

Overall posture: The register map is structurally sound — the address partitioning, FC allocation, big-endian convention, sentinel definitions, scaled-integer rationale, and versioning bump policy are all well-designed and internally complete where specified. The document's main weaknesses are systematic omissions (requirement citations and source/sink attribution on most registers) and four internal/cross-document contradictions that directly affect implementability. The RM-03 reserved-address contradiction and the RM-05 default-value contradiction will produce observable bugs in integration. Recommendation: do not approve for LLD start until RM-01 through RM-05 are resolved; RM-06 through RM-13 should be resolved in the same editing pass as they are quick fixes with high traceability value.2 / 2
