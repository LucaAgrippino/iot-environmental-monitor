# 05 · Project Setup

Two build targets share one source tree:

- **`sim`** — desktop SDL2 simulator built with **CMake**. Primary dev target for UI iteration.
- **`f469`** — STM32F469I-DISCO build managed by **STM32CubeIDE** (Eclipse).

The screens, models, and theme code are identical across both. Only the bootstrap (`main_sim.c` vs `main_f469.c`) and the persistence backend differ.

## Target board: STM32F469I-DISCO

Everything we need is on the board — no shield, no breakout:

| Peripheral | What's on board | Used for |
|---|---|---|
| MCU | STM32F469NIH6 (Cortex-M4, 180 MHz, FPU) | Everything |
| Flash | 2 MB internal | Code + emulated EEPROM (2 sectors at end) |
| RAM | 384 KB internal | Heap, stacks, FreeRTOS |
| External SDRAM | 16 MB IS42S32400F | LVGL framebuffer(s) |
| Display | 4" 800×480 MIPI-DSI LCD (FRD397B2509-D-CTM) | The UI |
| Touch | FT6206 capacitive, I²C2 | Input |
| Accelerator | Chrom-ART (DMA2D) | LVGL accelerated blits |
| Debug | Onboard ST-LINK/V2-1 | Flash + GDB |

The board's BSP (`stm32469i_discovery_*`) already initializes the LCD, DSI, and touch — use it instead of writing drivers from scratch.

## Hardware target: STM32CubeIDE setup

### 1. Create the project

In CubeIDE: **File → New → STM32 Project**, search for the **STM32F469I-DISCO** board, accept the default initialization to set up the BSP and the linker script.

### 2. CubeMX (`.ioc`) configuration

Open `envmon.ioc` in CubeIDE's graphical configurator. The important settings:

| Group | Setting | Value |
|---|---|---|
| **System Core → SYS** | Debug | Serial Wire |
| **System Core → SYS** | Timebase Source | TIM6 (free SysTick for FreeRTOS) |
| **System Core → RCC** | HSE | Crystal/Ceramic Resonator |
| **Clock Configuration** | HCLK | 180 MHz (PLL) |
| **Connectivity → I2C2** | Mode | I²C (touch controller) |
| **Connectivity → I2C2** | Pins | PB10/PB11 (board defaults) |
| **Multimedia → LTDC** | enabled by BSP | leave as BSP generates |
| **Multimedia → DSI Host** | enabled by BSP | leave as BSP generates |
| **Multimedia → DMA2D** | Mode | Register-to-Memory + Memory-to-Memory with PFC |
| **Connectivity → FMC** | SDRAM Bank 1 | enabled by BSP for the on-board IS42S32400F |
| **Middleware → FREERTOS** | Interface | CMSIS-OS v2 |
| **Middleware → FREERTOS** | Default task stack | 2048 words (LVGL needs it) |
| **Middleware → FREERTOS** | Heap | heap_4 |
| **Code Generator** | Generate peripheral init as pairs of `.c/.h` | yes |
| **Code Generator** | Keep User Code on regenerate | yes (default) |

### 3. Add LVGL

CubeIDE **Help → Manage Embedded Software Packages** has an LVGL pack (community-maintained). Install LVGL 9.x.

Then in the `.ioc`: **Middleware and Software Packs → LVGL → enable**. Set:

- `LV_COLOR_DEPTH` = 16
- `LV_HOR_RES_MAX` = 800
- `LV_VER_RES_MAX` = 480
- `LV_MEM_SIZE` = 96 KB
- `LV_USE_DMA2D` = 1 (Chrom-ART)
- `LV_USE_FLEX`, `LV_USE_GRID`, `LV_USE_CANVAS` = 1
- `LV_USE_LOG` = 1, `LV_LOG_LEVEL_WARN`
- `LV_FONT_DEFAULT` = `&font_inter_md_14` (custom; we provide)

If the pack version isn't available, just clone the LVGL 9 source into `Middlewares/Third_Party/lvgl/` and add the include + source paths in **Project → Properties → C/C++ Build → Settings → MCU GCC Compiler/Linker**.

### 4. LVGL port to the BSP

Two small files connect LVGL to the board:

**`lv_port/lv_port_disp.c`** — uses the BSP LCD layer. Two LVGL draw buffers in SDRAM (the BSP gives you SDRAM-mapped addresses); flush callback does a DMA2D copy (or `BSP_LCD_DrawBitmap` if you prefer). For an LVGL-9 reference, see ST's `STM32CubeF4/Projects/STM32469I-Discovery/Applications/LCD_DSI/`.

```c
// sketch — flesh out from the BSP examples
#include "lvgl.h"
#include "stm32469i_discovery_lcd.h"

#define DISP_W 800
#define DISP_H 480

// SDRAM-allocated frame buffer; LCD layer 0 already points here via the BSP.
static lv_color_t *fb1 = (lv_color_t *)LCD_FB_START_ADDRESS;
static lv_color_t  draw_buf[DISP_W * 60] __attribute__((section(".sdram"))); // partial buf

void lv_port_disp_init(void) {
    BSP_LCD_Init();
    BSP_LCD_LayerDefaultInit(0, (uint32_t)fb1);
    BSP_LCD_DisplayOn();

    lv_display_t *disp = lv_display_create(DISP_W, DISP_H);
    lv_display_set_buffers(disp, draw_buf, NULL, sizeof(draw_buf), LV_DISPLAY_RENDER_MODE_PARTIAL);
    lv_display_set_flush_cb(disp, flush_cb);
}
```

`flush_cb` should use **DMA2D** to copy `px_map` into the framebuffer; `lv_display_flush_ready()` is called from the DMA2D transfer-complete IRQ for non-blocking flush. This is critical for frame rate — a CPU `memcpy` won't keep up.

**`lv_port/lv_port_indev.c`** — wraps the BSP touch driver:

```c
#include "lvgl.h"
#include "stm32469i_discovery_ts.h"

static void touch_read(lv_indev_t *indev, lv_indev_data_t *data) {
    TS_StateTypeDef ts;
    BSP_TS_GetState(&ts);
    if (ts.touchDetected) {
        data->state   = LV_INDEV_STATE_PRESSED;
        data->point.x = ts.touchX[0];
        data->point.y = ts.touchY[0];
    } else {
        data->state = LV_INDEV_STATE_RELEASED;
    }
}

void lv_port_indev_init(void) {
    BSP_TS_Init(800, 480);
    lv_indev_t *t = lv_indev_create();
    lv_indev_set_type(t, LV_INDEV_TYPE_POINTER);
    lv_indev_set_read_cb(t, touch_read);
}
```

### 5. FreeRTOS task

In `Core/Src/freertos.c`, inside the `USER CODE` region for `defaultTaskFunction` (or a new task), drive LVGL:

```c
extern void app_init(void);

void StartLvglTask(void *argument) {
    lv_init();
    lv_port_disp_init();
    lv_port_indev_init();
    theme_init();
    app_init();
    for (;;) {
        osMutexAcquire(lvMutex, osWaitForever);
        uint32_t next = lv_timer_handler();
        osMutexRelease(lvMutex);
        osDelay(next < 5 ? 5 : next);
    }
}
```

`lvMutex` (a CMSIS-OS mutex) guards any UI access from other tasks. Any task that touches LVGL must take it.

### 6. Linker — SDRAM region

The BSP's startup + linker already exposes an SDRAM region. Confirm the linker script (`STM32F469NIHx_FLASH.ld`) places `.sdram` at `0xC0000000`. If you keep the LCD framebuffer at `LCD_FB_START_ADDRESS` (the BSP default), reserve a separate slab in SDRAM for LVGL canvases (sparklines) — see `04_LVGL_PATTERNS.md`.

### 7. Build & flash

CubeIDE handles this end-to-end: **Project → Build All**, then **Run → Run As → STM32 C/C++ Application** (ST-LINK appears automatically when the board is plugged in). Console output via SWO or the onboard ST-LINK VCP if you wire `printf`.

## Simulator target: CMake + SDL2

The simulator builds the same `src/` files on the host so the UI can be iterated without flashing.

**`target/sim/CMakeLists.txt`**:

```cmake
cmake_minimum_required(VERSION 3.20)
project(envmon_sim C)

set(CMAKE_C_STANDARD 11)
add_compile_options(-Wall -Wextra)

# LVGL — vendor it next to the sim target, or use a submodule
add_subdirectory(${CMAKE_SOURCE_DIR}/../../third_party/lvgl lvgl)
add_compile_definitions(LV_CONF_PATH="${CMAKE_CURRENT_SOURCE_DIR}/lv_conf_sim.h")

find_package(SDL2 REQUIRED)

# Glob shared sources, exclude target-specific bits
file(GLOB_RECURSE SHARED_SRC
    ${CMAKE_SOURCE_DIR}/../../src/*.c
)
list(FILTER SHARED_SRC EXCLUDE REGEX "persistence_eeprom\\.c$")

add_executable(envmon_sim
    main_sim.c
    ${SHARED_SRC}
)
target_include_directories(envmon_sim PRIVATE
    ${CMAKE_SOURCE_DIR}/../../src
    ${SDL2_INCLUDE_DIRS}
)
target_link_libraries(envmon_sim PRIVATE lvgl SDL2::SDL2)
```

**`target/sim/main_sim.c`** — uses LVGL's bundled SDL driver (`LV_USE_SDL 1` in `lv_conf_sim.h`):

```c
#include <SDL2/SDL.h>
#include "lvgl.h"
#include "app.h"

int main(int argc, char *argv[]) {
    lv_init();
    lv_sdl_window_create(800, 480);
    lv_sdl_mouse_create();
    theme_init();
    app_init();
    while (1) {
        lv_timer_handler();
        SDL_Delay(5);
    }
}
```

**`target/sim/lv_conf_sim.h`** is a copy of the hardware `lv_conf.h` with these overrides:

- `LV_COLOR_DEPTH 32` (SDL wants ARGB8888)
- `LV_USE_DMA2D 0` (no Chrom-ART on the host)
- Drop the `LV_ATTRIBUTE_FAST_MEM` definition (no IRAM on the host)

Build:

```bash
brew install sdl2                # or: apt install libsdl2-dev
cd target/sim
cmake -B build
cmake --build build
./build/envmon_sim
```

## Persistence — emulated EEPROM on F469

STM32 has no NVS. Use ST's **X-CUBE-EEPROM** application note **AN3969** library: it dedicates two flash sectors (we'll use the last two of the 2 MB flash) and does wear-levelling for ≤ 4 KB of key/value data.

In CubeIDE:

1. Download X-CUBE-EEPROM from ST (or copy the `eeprom_emul/` source from `STM32CubeF4` examples).
2. Add `eeprom_emul.c/.h` and `flash_interface.c/.h` to the project.
3. Reserve sectors 22 and 23 (the last two 128 KB sectors on the F469) in the linker by editing `STM32F469NIHx_FLASH.ld`:
   ```
   FLASH    (rx)    : ORIGIN = 0x08000000, LENGTH = 1792K   /* sectors 0..21 */
   EE_FLASH (r)     : ORIGIN = 0x081C0000, LENGTH = 256K    /* sectors 22-23 */
   ```
4. `EE_Init()` runs at boot, before LVGL needs config values.

`persistence_eeprom.c` wraps this behind the same `persistence.h` interface used by the simulator's file backend — see `06_DATA_MODEL.md`. The set of keys is small (~8 fields × 4 bytes each) so EEPROM emulation is comfortable for our write frequency (only on user "Save").

## Performance budgets

| Metric | Target | Notes |
|---|---|---|
| Splash → first sensor screen | ≤ 3 s | LVGL init + first poll |
| Tab switch | < 80 ms on F469 | Free old screen, create new |
| Sensor card refresh (per card) | < 5 ms | Easy at 5 s poll |
| Free heap at steady state | ≥ 32 KB | Fail boot if below at hand-off |
| Frame rate (steady) | 30 FPS | Only sparklines + blink animate; DMA2D handles it |

If frame rate dips, the first thing to check is whether `flush_cb` is using DMA2D and signalling completion from the IRQ (vs. a blocking copy on the LVGL task).
